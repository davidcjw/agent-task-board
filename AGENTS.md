# Agent Task Board — agent guide

A local-first kanban (Next.js 16 + React 19 + Tailwind v4) for tracking tasks delegated to AI coding agents. Four lanes: Queued → Running → Review → Done. State lives entirely in `localStorage`.

> Note: this codebase already follows Next 16's strict React Compiler lint rules — **no ref writes during render, no synchronous `setState` inside an effect body**. The board state uses `useSyncExternalStore` partly for this reason.

## Commands

```bash
npm run dev         # dev server (Turbopack) on :3000 (serves UI + /api routes)
npm run build       # production build (must pass before shipping)
npm run lint        # ESLint — strict; must be clean
npm run typecheck   # tsc --noEmit — must be clean
npm run test        # Vitest unit suite — must be green
npm run dispatcher  # agent dispatcher (dry-run unless --execute)
npm run watcher     # merge-watcher: Review → Done when a card's PR is merged (polls gh)
npm run telegram    # inbound Telegram bot (needs TELEGRAM_BOT_TOKEN)
npm run mcp         # MCP stdio server exposing board tools
npm run agents      # control plane: board (api mode) + dispatcher + merge-watcher; one Ctrl-C stops all
npm run agents:install / agents:uninstall   # macOS LaunchAgent: run the whole control plane persistently
```

Always run `lint`, `typecheck`, `test`, and `build` before considering a change done.

## Architecture

- **Pure logic lives in `lib/`** and is framework-free and unit-tested. `board.ts` is the reducer (`addTask`, `updateTask`, `deleteTask`, `moveTask`, `commitDrag`, `claimNext`, `setResult`, `reconcile`); ids and timestamps are injectable so functions are deterministic. Add tests alongside any logic change (`lib/*.test.ts`).
- **Two engines, one contract** (`lib/boardEngine.ts`): `localEngine` (localStorage) and `apiEngine` (server-backed, polls `/api/board`). `lib/useBoard.ts` picks one by `NEXT_PUBLIC_BOARD_MODE` (`local` default | `api`) and exposes it via `useSyncExternalStore`. The `EMPTY` sentinel is both the SSR snapshot and the "not hydrated" marker. **Both engines drive the same `board.ts` reducer**, so local and live behave identically — keep it that way.
- **`BoardState`** = a flat `tasks` map + ordered id-lists per column (`columns: Record<Status, string[]>`). Reorders/moves are array splices; drag uses dnd-kit `arrayMove`.
- **Server** is in `lib/server/` (server-only — never import from client components): `store.ts` is a JSON-file board with an in-process async **mutex** so claims are atomic (one task → one agent). Route Handlers in `app/api/` (board, tasks, claim, result) wrap it.
- **Agent layer** in `agent/` (plain Node ESM, run via npm scripts): `dispatcher.mjs` (claim → route by `agent` label via `routes.json` → run → report to board + Telegram), `mcp-server.mjs` (board tools over stdio), `telegram-bot.mjs` (messages → tasks), `lib/api.mjs` + `lib/telegram.mjs`. **The dispatcher is dry-run by default**; `--execute` / `AGENT_EXECUTE=1` actually runs runner commands. **Result lane** (pure `resultStatus` in `lib/routes.mjs`): a task that opened a PR — or that errored, or any dry-run preview — lands in **Review** (the human gate; the merge-watcher later advances a merged PR to Done); a task that opened **no PR** (a plain question, info/subagent work, or a code run with no diff) skips Review and goes **straight to Done**.
  - **Routing** (`agent/routes.json`, gitignored — copy `routes.example.json` to start; falls back to the example if absent): each entry maps an `agent` label to a runner `{command, args, cwd}`. `args`/`cwd` strings interpolate `{prompt}` `{title}` `{id}` `{agent}` `{tags}`. A route's `cwd` of `"{repo}"` makes **one route serve every repo**: the task's `repo:<name>` tag resolves to `<AGENT_REPO_BASE>/<name>` (default `~/code`; absolute tag values win), with a missing tag falling back to the dispatcher's own cwd (logged). Code routes (`default`, `Claude Code`) use `"{repo}"` + `"pr": true`; subagent routes (`knowledge-base`, `course-admin`, `renovation-advisor`) run `claude --agent <name>` with a literal `cwd: "."` and no `pr`. **`pr: true` + a `repo:` tag** triggers the **dispatcher-driven PR flow** (`shouldOpenPr`): the agent gets an *edit-only* prompt (`implementPrompt` — "make code changes, don't touch git"), then the **dispatcher itself** (`lib/git.mjs`: `createWorktree` → run agent → `finishPr` → `removeWorktree`) commits, pushes, and runs `gh pr create --fill`, appending `BOARD_PR: <url>` to the result so the task arrives in Review *already a PR*. It's deterministic (no reliance on the model running git) and **isolated**: each task runs in its own `git worktree` (branch `atb/<id>`, under `AGENT_WORKTREE_DIR`, default `$TMPDIR/atb-worktrees`), so it never touches your main checkout and **concurrent tasks — even on the same repo — don't collide**. The fresh worktree is hydrated with a `node_modules` symlink + copied `.env`/`.env.local` so builds/tests still work; worktree creation/removal is serialized per repo. **`finishPr` unstages those hydration artifacts before committing** (`git add -A` → `git reset -- node_modules .env .env.local`) so they never leak into the PR — a `node_modules` *symlink* slips past a `node_modules/` (dir-only) `.gitignore` rule, and a copied `.env` would leak in a repo that doesn't ignore it. A task that produces no diff opens no PR; tasks without a `repo:` tag (plain questions, subagent work) never PR. This folds in the old `commit-push` route. Pure routing helpers are the unit-tested `lib/routes.mjs` (`repoFromTags`, `resolveRepoPath`, `resolveCwd`, `missingRepoTag`, `shouldOpenPr`, `branchName`, `implementPrompt`, `worktreePath`); the `git`/`gh` shelling lives in `lib/git.mjs` (impure, like `prs.mjs`).
  - **Review gate** (opt-in, `lib/review.mjs`): before `finishPr` opens the PR, a `pr:true` route can run an **independent pre-PR review loop** in the same worktree. A *fresh* agent process (no implementer context — genuinely independent, since every `claude -p` is one-shot) reviews the diff alongside the repo's **own checks** (auto-detected `lint`/`typecheck`/`test` scripts, run via `npm run`), and a fixer process iterates until the change clears the gate or a cap is hit. **The gate passes only when all three hold: checks green AND zero blocking findings AND reviewer confidence ≥ threshold** (default 95) — the soft confidence signal is anchored to the hard (checks) and concrete (blocking) ones, because an LLM's confidence number isn't calibrated. If the cap (default 2 fix rounds) is reached without passing, the PR **still opens but flagged** (findings + confidence folded into `task.result`, so the card + Telegram show `⚠ Flagged`). Enabled per route via `"review": true` or `"review": { iterations, threshold, checks }`, or forced for every PR route with `AGENT_REVIEW=1` (`=0` off). Pure helpers (`shouldReview`, `reviewConfig`, `detectChecks`, `parseReview` (fails closed), `reviewVerdict`, `reviewPrompt`/`fixPrompt`, `reviewSummary`) are unit-tested in `lib/review.test.mjs`; the impure `runChecks`/`runReviewer`/`reviewLoop` shell out (reusing `git.mjs`'s exported `run` + new `worktreeDiff`) and take the dispatcher's `runCommand` injected. `no-mistakes` was evaluated and deliberately **not** integrated (heavyweight push-proxy daemon; too much overlap).
  - **Concurrency:** the dispatcher runs one task at a time by default; `--concurrency N` / `AGENT_CONCURRENCY` (forwarded by `npm run agents -- --concurrency N`) runs up to N at once via a worker pool. Safe because each PR task is worktree-isolated. The server's claim mutex already guarantees one-task-one-agent, so running multiple dispatcher processes also works.
  - `launch.mjs` (`npm run agents`) is the **process manager** that ties the layer together: it spawns the board in `api` mode, polls `/api/board` until ready, then starts the dispatcher + merge-watcher (and the inbound bot), prefixing each child's output and tearing them all down on Ctrl-C or if any one exits. The built-in bot is **auto-on whenever `TELEGRAM_BOT_TOKEN` is set** (every message → a queued task; parsing in `lib/message.mjs`); pass `--no-telegram` to disable it — do that when an external front door (e.g. hans) owns inbound **on the same bot**, since two `getUpdates` pollers on one token → `409 Conflict` (the dispatcher only *sends*, so it never collides). Give the board its own bot token to run both. Flags: `--execute`, `--no-telegram`, `--prod`, `--no-board`, `--no-watcher`. **Inbound auth:** the bot only queues tasks from chats in `ALLOWED_CHAT_IDS` (comma-separated), falling back to `TELEGRAM_CHAT_ID`; non-allowlisted chats get a refusal (`/id` stays open for discovery). With neither set it stays open but warns at startup — trusted-network mode, mirroring the worker endpoints' `AGENT_TOKEN` posture. **Repo selection** (faster than typing `#repo:<name>`): a leading `/slug` names the repo for one task (`/democratizing_claude fix login`), and `/use <repo>` sets a per-chat sticky default so plain messages inherit it (`/use` shows it, `/use off` clears it; stored in `.data/tg-repos.json`). A bare `/slug` with no task switches the sticky repo. Slugs are matched **separator-insensitively** against the dirs under `AGENT_REPO_BASE` (so the underscore-only `/democratizing_claude` Telegram allows resolves to `democratizing-claude`) via pure `matchRepoSlug`/`normalizeRepoKey` in `lib/routes.mjs` over `listRepos` in `lib/repos.mjs`; on startup the bot registers each repo (+ `/use`,`/id`,`/help`) as a `/`-autocomplete command (`setMyCommands`, `repoCommandName`). Precedence: `/slug` > explicit `#repo:` tag > sticky default.
  - `launchd/install.mjs` (`npm run agents:install`) writes + loads a macOS LaunchAgent (`KeepAlive`/`RunAtLoad`) that runs the **whole control plane** persistently via `launch.mjs` (board + dispatcher + watcher + bot) — it provides its own board, so nothing else need run (don't also `npm run agents` by hand: two boards clash on port 3000). It widens `PATH` so runners find `claude`/`gh`. Forwards `--execute`, `--prod` (builds first), `--no-telegram`, `--no-watcher`; `--print` previews the plist; logs to `.data/controlplane.{out,err}.log`. Label `com.davidcjw.agent-task-board.controlplane`.
  - `merge-watcher.mjs` (`npm run watcher`, started by `npm run agents` unless `--no-watcher`) closes the loop: each sweep it scans **Review** cards, extracts a `github.com/.../pull/N` url from `task.result` (pure helper in `lib/prs.mjs`, unit-tested in `lib/prs.test.mjs`), asks `gh` if that PR is merged, and if so moves the card to **Done** + notifies Telegram. It only shells out to `gh` when a Review card actually carries a PR url, so it's silent until there's something to watch. Interval: `WATCHER_INTERVAL` ms (default 30000) or `--interval`. Those PRs are opened by the dispatcher-driven PR flow (`pr: true` code route + `repo:` tag) which writes `BOARD_PR: <url>` into the task result; see Routing above. Note: `agent/**/*.test.mjs` is now in the vitest `include`.
- **UI** is in `components/`. Orchestrator `BoardApp.tsx`; drag-and-drop in `Board.tsx`; prompt-first card `TaskCard.tsx` (also renders the agent result block, the dedicated PR-link field, and the compact one-line form for the Done lane). **Archive**: `Task.archivedAt` (set via the `archived` flag on `updateTask` — threaded through both engines + the PATCH route) hides a card from its lane; archived cards render behind a per-lane "show archived" reveal in `Column.tsx`. `lib/board.ts` exposes `isArchived`; `BoardApp` filters archived out of the lanes and lane counts.

## Design system

The visual language is **vendored** from `~/code/design-systems/dragonfly-ds` into `components/ds/` (primitives `Panel`, `Text`, `Button`, `Rule` + `dragonfly.css` tokens). Do not hand-edit `components/ds/dragonfly.css` — re-vendor from source if the DS changes. Fonts (Fraunces / Inter / JetBrains Mono) are self-hosted via `next/font` and mapped onto the `--df-font-*` tokens in `app/globals.css`. Stay faithful to the dark editorial look: black canvas, hairline grid, single orange accent (`#fa4c14`), monospace labels. The four lane status colours live in `app/globals.css` (`@theme`) and `components/status.ts`.

## Conventions

- No emoji as icons — use the inline SVGs in `components/icons.tsx`.
- Tailwind class names must be literal (not interpolated) so JIT picks them up; per-status class maps live in `components/status.ts`.
- Keep it local-first: nothing should make a network request. No analytics, no backend.
