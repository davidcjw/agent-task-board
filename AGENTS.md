# Agent Task Board — agent guide

A local-first kanban (Next.js 16 + React 19 + Tailwind v4) for tracking tasks delegated to AI coding agents. Four lanes: Queued → Running → Review → Done. State lives entirely in `localStorage`.

> Note: this codebase already follows Next 16's strict React Compiler lint rules — **no ref writes during render, no synchronous `setState` inside an effect body**. The board state uses `useSyncExternalStore` partly for this reason.

## Commands

```bash
npm run dev         # dev server (Turbopack) on :3000 (serves UI + /api routes)
npm run build       # production build (must pass before shipping)
npm run lint        # ESLint — strict; must be clean
npm run typecheck   # tsc --noEmit — must be clean
npm run test        # Vitest unit suite — must be green
npm run dispatcher  # agent dispatcher (dry-run unless --execute)
npm run watcher     # merge-watcher: Review → Done when a card's PR is merged (polls gh)
npm run history     # run-history stats CLI: summarize .data/history.jsonl (--json for raw)
npm run telegram    # inbound Telegram bot (needs TELEGRAM_BOT_TOKEN)
npm run mcp         # MCP stdio server exposing board tools
npm run agents      # control plane: board (api mode) + dispatcher + merge-watcher; one Ctrl-C stops all
npm run agents:install / agents:uninstall   # macOS LaunchAgent: run the whole control plane persistently
npm run scout       # improvement scout: scan ~/code, rank ideas, queue the single best (--dry-run to preview)
npm run scout:install / scout:uninstall     # macOS LaunchAgent: run the scout every 2h, 8am–10pm
```

Always run `lint`, `typecheck`, `test`, and `build` before considering a change done.

## Architecture

- **Pure logic lives in `lib/`** and is framework-free and unit-tested. `board.ts` is the reducer (`addTask`, `updateTask`, `deleteTask`, `moveTask`, `commitDrag`, `claimNext`, `setResult`, `revisePatch`, `reconcile`); ids and timestamps are injectable so functions are deterministic. `setResult` also persists the runner's `Task.sessionId`; `revisePatch` builds the Review→Queued send-back patch (see the Revise flow below). Add tests alongside any logic change (`lib/*.test.ts`).
- **Two engines, one contract** (`lib/boardEngine.ts`): `localEngine` (localStorage) and `apiEngine` (server-backed, polls `/api/board`). `lib/useBoard.ts` picks one by `NEXT_PUBLIC_BOARD_MODE` (`local` default | `api`) and exposes it via `useSyncExternalStore`. The `EMPTY` sentinel is both the SSR snapshot and the "not hydrated" marker. **Both engines drive the same `board.ts` reducer**, so local and live behave identically — keep it that way.
- **`BoardState`** = a flat `tasks` map + ordered id-lists per column (`columns: Record<Status, string[]>`). Reorders/moves are array splices; drag uses dnd-kit `arrayMove`.
- **Server** is in `lib/server/` (server-only — never import from client components): `store.ts` is a JSON-file board with an in-process async **mutex** so claims are atomic (one task → one agent). Route Handlers in `app/api/` (board, tasks, claim, result) wrap it. **Auth is uniform** (`lib/server/auth.ts`): with `AGENT_TOKEN` unset the API is open (local-first default); when set, **every** `/api` route requires `Authorization: Bearer <token>` and 401s otherwise (constant-time compare via `timingSafeEqual`; route-level tests in `lib/server/auth.test.ts`). Any new Route Handler must start with the same `authorized(req)` check. Clients: `agent/lib/api.mjs` (and `launch.mjs`'s readiness poll) send `AGENT_TOKEN`; the browser `apiEngine` sends `NEXT_PUBLIC_AGENT_TOKEN` (build-time-inlined into the bundle — readable by anyone served the UI, so it's a trusted-network convenience, not a secret).
- **Agent layer** in `agent/` (plain Node ESM, run via npm scripts): `dispatcher.mjs` (claim → route by `agent` label via `routes.json` → run → report to board + Telegram), `mcp-server.mjs` (board tools over stdio), `telegram-bot.mjs` (messages → tasks), `lib/api.mjs` + `lib/telegram.mjs`. **The dispatcher is dry-run by default**; `--execute` / `AGENT_EXECUTE=1` actually runs runner commands. **Result lane** (pure `resultStatus` in `lib/routes.mjs`): a task that opened a PR — or that errored, or any dry-run preview — lands in **Review** (the human gate; the merge-watcher later advances a merged PR to Done); a task that opened **no PR** (a plain question, info/subagent work, or a code run with no diff) skips Review and goes **straight to Done**.
  - **Routing** (`agent/routes.json`, gitignored — copy `routes.example.json` to start; falls back to the example if absent): each entry maps an `agent` label to a runner `{command, args, cwd}`. `args`/`cwd` strings interpolate `{prompt}` `{title}` `{id}` `{agent}` `{tags}`. A route's `cwd` of `"{repo}"` makes **one route serve every repo**: the task's `repo:<name>` tag resolves to `<AGENT_REPO_BASE>/<name>` (default `~/code`; absolute tag values win), with a missing tag falling back to the dispatcher's own cwd (logged). Code routes (`default`, `Claude Code`) use `"{repo}"` + `"pr": true`; subagent routes (`knowledge-base`, `course-admin`, `renovation-advisor`) run `claude --agent <name>` with a literal `cwd: "."` and no `pr`. **`pr: true` + a `repo:` tag** triggers the **dispatcher-driven PR flow** (`shouldOpenPr`): the agent gets an *edit-only, worktree-confined* prompt (`implementPrompt` — "make code changes in the current working directory, don't touch git"). That prompt is also the **backstop against worktree escapes**: it tells the agent to work ONLY inside its cwd (the isolated worktree), never `cd`, and — critically — to **ignore any absolute path in the task** (e.g. `/Users/…/code/<repo>/…`) and act on the matching *worktree-relative* path instead. Without this, a `--dangerously-skip-permissions` agent obeys an absolute path embedded in the task (e.g. by the scout) over its worktree cwd, editing/committing in the developer's **real** checkout — which leaves the worktree empty (no PR, task skips Review straight to Done) while stray commits pile up under `~/code`, and each dirtied checkout makes the next incremental scan propose "commit the pending workflow" → a self-reinforcing escape loop. The scout's `scoutPrompt` is the matching fix at the source (repo-relative paths only, never `cd`, no git steps). After the agent finishes, the **dispatcher itself** (`lib/git.mjs`: `createWorktree` → run agent → `finishPr` → `removeWorktree`) commits, pushes, and runs `gh pr create --fill`, appending `BOARD_PR: <url>` to the result so the task arrives in Review *already a PR*. It's deterministic (no reliance on the model running git) and **isolated**: each task runs in its own `git worktree` (branch `atb/<id>`, under `AGENT_WORKTREE_DIR`, default `$TMPDIR/atb-worktrees`), so it never touches your main checkout and **concurrent tasks — even on the same repo — don't collide**. The fresh worktree is hydrated with a `node_modules` symlink + copied `.env`/`.env.local` so builds/tests still work; worktree creation/removal is serialized per repo. **`finishPr` unstages those hydration artifacts before committing** (`git add -A` → `git reset -- node_modules .env .env.local`) so they never leak into the PR — a `node_modules` *symlink* slips past a `node_modules/` (dir-only) `.gitignore` rule, and a copied `.env` would leak in a repo that doesn't ignore it. A task that produces no diff opens no PR; tasks without a `repo:` tag (plain questions, subagent work) never PR. This folds in the old `commit-push` route. Pure routing helpers are the unit-tested `lib/routes.mjs` (`repoFromTags`, `resolveRepoPath`, `resolveCwd`, `missingRepoTag`, `shouldOpenPr`, `branchName`, `implementPrompt`, `worktreePath`); the `git`/`gh` shelling lives in `lib/git.mjs` (impure, like `prs.mjs`).
  - **Review gate** (opt-in, `lib/review.mjs`): before `finishPr` opens the PR, a `pr:true` route can run an **independent pre-PR review loop** in the same worktree. A *fresh* agent process (no implementer context — genuinely independent, since every `claude -p` is one-shot) reviews the diff alongside the repo's **own checks** (auto-detected `lint`/`typecheck`/`test` scripts, run via `npm run`), and a fixer process iterates until the change clears the gate or a cap is hit. **The gate passes only when all three hold: checks green AND zero blocking findings AND reviewer confidence ≥ threshold** (default 90) — the soft confidence signal is anchored to the hard (checks) and concrete (blocking) ones, because an LLM's confidence number isn't calibrated. If the cap (default 1 fix round → at most 2 review passes) is reached without passing, the PR **still opens but flagged** (findings + confidence folded into `task.result`, so the card + Telegram show `⚠ Flagged`). The Telegram review notice drops the file-by-file *Changes* section but always keeps the `🔍 Review:` verdict block in full (pure `notifyBody` in `lib/message.mjs`, unit-tested). Enabled per route via `"review": true` or `"review": { iterations, threshold, checks }`, or forced for every PR route with `AGENT_REVIEW=1` (`=0` off). Pure helpers (`shouldReview`, `reviewConfig`, `detectChecks`, `parseReview` (fails closed), `reviewVerdict`, `reviewPrompt`/`fixPrompt`, `reviewSummary`) are unit-tested in `lib/review.test.mjs`; the impure `runChecks`/`runReviewer`/`reviewLoop` shell out (reusing `git.mjs`'s exported `run` + new `worktreeDiff`) and take the dispatcher's `runCommand` injected. `no-mistakes` was evaluated and deliberately **not** integrated (heavyweight push-proxy daemon; too much overlap).
  - **Revise / send-back (Review → Queued, same PR):** a Review card whose PR has failing CI or a merge conflict can be sent back for another pass that **updates the same PR** instead of opening a new one. Three parts. **(1) Session capture:** every `claude --output-format json` run reports a `session_id`; `runCommand` now keeps it and `runWithPr`/`runWithRevise` carry the *implementer's* id up to the board as `Task.sessionId` (threaded through `reportResult` → the result route → `setResult`). The session transcript lives under `~/.claude` keyed by the worktree **path** (deterministic by task id via `worktreePath`), so it survives the worktree teardown and resumes when a new worktree is recreated at the same path — verified: resume recalls context even after the dir is deleted and recreated. **(2) Board action** (`lib/board.ts`): the pure `revisePatch(task, note)` builds a `TaskPatch` that returns the card to Queued, adds the `revise` tag (`REVISE_TAG`, deduped by `updateTask`'s tag normalization), and stamps the human's correction as `reviseNote` — leaving `prompt`, `result`, `sessionId`, and `createdAt` intact (so the original session resumes and FIFO `claimNext` re-picks it next). It rides the existing `updateTask`/PATCH rails (no new engine method/endpoint); `coerceTaskPatch` carries `reviseNote`. In the UI a **"Send back for revision"** button shows only on Review cards (`RewindIcon`), opening `ReviseDialog` to capture the note; `BoardApp.handleRevise` applies it with a toast + undo. **(3) Dispatcher revise mode:** when a claimed task is `opensPr && isRevise(task)`, `processTask` routes to `runWithRevise` instead of `runWithPr`. It reopens the **existing** PR branch — `createReviseWorktree` (`lib/git.mjs`) does `git fetch` → `git worktree add -B <branch> origin/<branch>` at the same path → hydrate → and **merges the latest base in only when the branch is behind** (`merge-base --is-ancestor`); a conflicting merge is left in the tree for the agent, surfaced as `mergeConflict`. The agent runs with `revisePrompt(task, {mergeConflict})` (the correction note + original task + the same edit-only/worktree-confinement backstop as `implementPrompt`, plus conflict-resolution instructions when needed) and, when a `sessionId` exists, **resumes its original session** via `resumeRoute` (clones the `claude` route to add `--resume <id>`; falls back to a fresh agent for a missing id or a non-resumable route — the checked-out branch already gives it the code context). `finishRevise` stages (dropping hydration artifacts), **refuses while conflict markers remain** (`git diff --cached --check`), completes the merge or commits the fixes, **fast-forward-pushes** to the same branch (merge, not rebase → no force-push), and resolves the PR url — the PR updates in place. Cancel/timeout bail before committing, like `runWithPr`; failures (missing origin branch, non-conflict merge failure, lingering markers) land back in Review flagged. **Merge, not rebase** is deliberate: no force-push, one conflict-resolution point, fast-forward push — far more robust for an automated agent flow (the PR gets a merge commit). Pure helpers (`isRevise`, `resumeRoute`, `revisePrompt`) are unit-tested in `lib/routes.test.mjs`; the impure `createReviseWorktree`/`finishRevise` shell out. **Telegram `/revise`** (control-plane bot): bare `/revise` lists the revise-capable Review cards (those carrying an open PR); `/revise <id-prefix> <correction>` sends the matched card back — the bot `PATCH`es the same `revisePatch` shape (`api.patchTask`) so board-UI and Telegram share one path. Pure helpers (`parseReviseCommand`, `reviseCandidates`, `matchReviseTask`, `reviseLabel`, `reviseListText`, `revisePatch`) live in `lib/revise.mjs`, unit-tested in `lib/revise.test.mjs`; registered via `setMyCommands`. Like `/scout`, it only works when the control-plane bot owns inbound (`getUpdates`). **Finding the id:** the dispatcher's Telegram notice for a PR landing in Review appends a ready-to-use `↩️ send back: /revise <id8> <correction>` line, and `prBody` writes a `_Board task: <id>_` footer into the PR body (GitHub ↔ board mapping) — so you rarely need to hunt for it.
  - **Concurrency:** the dispatcher runs one task at a time by default; `--concurrency N` / `AGENT_CONCURRENCY` (forwarded by `npm run agents -- --concurrency N`) runs up to N at once via a worker pool. Safe because each PR task is worktree-isolated. The server's claim mutex already guarantees one-task-one-agent, so running multiple dispatcher processes also works.
  - **Timeout & auto-requeue:** every runner is bounded by `AGENT_TIMEOUT` (default 20 min); on overrun the dispatcher SIGKILLs it (`runCommand` sets `timedOut`). A timed-out task is **auto-requeued once** — moved back to Queued, stamped with the `AUTO_RETRY_TAG` (`"auto-retry"`), and announced on Telegram. The tag makes it **one-shot** (pure `shouldRequeue` in `lib/routes.mjs`, unit-tested): a second timeout falls through to the normal Review-flagged-`❌` path for a human. The PR flow **bails before committing when the implementer times out** (`runWithPr` returns early on `agent.timedOut`), so the discarded worktree never pushes a partial branch and the clean retry can't collide. Only timeouts retry — genuine non-zero exits and start errors go straight to Review (re-running them just burns another `AGENT_TIMEOUT`). Auto-requeue is scoped to the **implementer** run; reviewer/fixer/check timeouts inside the review gate don't trigger it. The requeue keeps the task's original `createdAt`, so FIFO `claimNext` re-picks it next. See the README's *Timeouts & failure handling* table for the full scenario matrix.
  - **Cancellation (Telegram `/cancel`):** you can stop an in-flight task from Telegram. Because the bot and dispatcher are **separate processes**, cancellation is a *signal routed through the board*, not a direct kill: `POST /api/tasks/:id/cancel` (`store.requestCancel`, only valid while `running`) stamps `cancelRequestedAt`; the dispatcher runs an independent **cancel-poll** timer (`AGENT_CANCEL_POLL`, default 3s — its own timer, so it fires even in sequential mode where the claim loop is blocked) that aborts the matching task's `AbortController`. Each task gets one controller (`active` map keyed by id), and **every** child spawned for it — the agent run, the review checks (`git.mjs run`), the reviewer/fixer — is spawned **detached** and honors the signal, so an abort **process-group-kills** the whole tree (`killGroup` in `lib/git.mjs`: `process.kill(-pid)`; this also upgrades the timeout kill, which used a child-only `SIGKILL` that could orphan `claude`'s sub-processes). A cancelled run **moves straight to Done** with a `🛑 Cancelled by you` result and is **never auto-requeued** (`processTask` checks `signal.aborted` *before* the `shouldRequeue` branch); the PR flow **bails before commit/push** (`runWithPr` returns `canceled` on `agent.canceled`/`signal.aborted`), so the worktree — torn down in `finally` — never pushes a partial branch. **Concurrency:** the signal is per-task on the board, so the right worker reacts and others are untouched; the Telegram UX disambiguates — bare `/cancel` cancels the sole running task or shows an **inline picker** when several run, plus `/cancel <id-prefix>` and `/cancel all` (pure helpers `cancelKeyboard`/`parseCancelCallback`/`matchRunningTask`/`runningLabel` in `lib/cancel.mjs`, unit-tested; the bot's `handleCallback` fields the button taps). The dispatcher also aborts all in-flight runs on `SIGTERM`/`SIGINT`, so a control-plane Ctrl-C no longer orphans agents.
  - `launch.mjs` (`npm run agents`) is the **process manager** that ties the layer together: it spawns the board in `api` mode, polls `/api/board` until ready, then starts the dispatcher + merge-watcher (and the inbound bot), prefixing each child's output and tearing them all down on Ctrl-C or if any one exits. The built-in bot is **auto-on whenever `TELEGRAM_BOT_TOKEN` is set** (every message → a queued task; parsing in `lib/message.mjs`); pass `--no-telegram` to disable it — do that when an external front door (e.g. hans) owns inbound **on the same bot**, since two `getUpdates` pollers on one token → `409 Conflict` (the dispatcher only *sends*, so it never collides). Give the board its own bot token to run both. Flags: `--execute`, `--no-telegram`, `--prod`, `--no-board`, `--no-watcher`. **Inbound auth:** the bot only queues tasks from chats in `ALLOWED_CHAT_IDS` (comma-separated), falling back to `TELEGRAM_CHAT_ID`; non-allowlisted chats get a refusal (`/id` stays open for discovery). With neither set it stays open but warns at startup — trusted-network mode, mirroring the worker endpoints' `AGENT_TOKEN` posture. **Repo selection** (faster than typing `#repo:<name>`): a leading `/slug` names the repo for one task (`/democratizing_claude fix login`), and `/use <repo>` sets a per-chat sticky default so plain messages inherit it (`/use` shows it, `/use off` clears it; stored in `.data/tg-repos.json`). A bare `/slug` with no task switches the sticky repo. Slugs are matched **separator-insensitively** against the dirs under `AGENT_REPO_BASE` (so the underscore-only `/democratizing_claude` Telegram allows resolves to `democratizing-claude`) via pure `matchRepoSlug`/`normalizeRepoKey` in `lib/routes.mjs` over `listRepos` in `lib/repos.mjs`; on startup the bot registers each repo (+ `/use`,`/cancel`,`/scout`,`/id`,`/help`) as a `/`-autocomplete command (`setMyCommands`, `repoCommandName`). Precedence: `/slug` > explicit `#repo:` tag > sticky default.
  - `launchd/install.mjs` (`npm run agents:install`) writes + loads a macOS LaunchAgent (`KeepAlive`/`RunAtLoad`) that runs the **whole control plane** persistently via `launch.mjs` (board + dispatcher + watcher + bot) — it provides its own board, so nothing else need run (don't also `npm run agents` by hand: two boards clash on port 3000). It widens `PATH` so runners find `claude`/`gh`. Forwards `--execute`, `--prod` (builds first), `--no-telegram`, `--no-watcher`; `--print` previews the plist; logs to `.data/controlplane.{out,err}.log`. Label `com.davidcjw.agent-task-board.controlplane`.
  - **Run history** (`lib/history.mjs`, CLI `history.mjs` = `npm run history`): every finished task (success/failure/cancel path where the dispatcher reports back to the board) appends one JSONL line to **`.data/history.jsonl`** — `historyRecord({id,title,agent,repo,status,durationMs,reviewScore,prUrl,error,at})` normalizes it, `appendHistory` writes it best-effort (swallows fs errors, returns a bool like `pending.mjs`; a history write **never** breaks the dispatch loop). The dispatcher captures a start timestamp when it claims a task, and folds in the review-gate confidence (`reviewScore`) and PR url when present. `readHistory` parses the log (skipping malformed lines) and the **pure** `summarizeHistory` returns `{total, byStatus, successRate, avgDurationMs, byRepo, byAgent}`, which `npm run history` prints (or `--json` dumps); an empty/absent log prints a clean zero-state. Pure helpers unit-tested in `lib/history.test.mjs`. The `.data/` gitignore rule already covers the log.
  - `merge-watcher.mjs` (`npm run watcher`, started by `npm run agents` unless `--no-watcher`) closes the loop: each sweep it scans **Review** cards, extracts a `github.com/.../pull/N` url from `task.result` (pure helper in `lib/prs.mjs`, unit-tested in `lib/prs.test.mjs`), asks `gh` for that PR's state, and acts on the outcome: **merged** → move the card to **Done**; **closed without merging** (you rejected it) → **delete the card** (`isClosed` in `lib/prs.mjs` + `api.deleteTask`). Either way it also **prunes the stale `atb/<id>` head branch** (`deleteRemoteBranch` + `repoFromPrUrl`; `prState` now returns `headBranch`; guarded to `atb/*` so a hand-made PR's branch is never touched). Both notify Telegram. It only shells out to `gh` when a Review card actually carries a PR url, so it's silent until there's something to watch. Interval: `WATCHER_INTERVAL` ms (default 30000) or `--interval`. Those PRs are opened by the dispatcher-driven PR flow (`pr: true` code route + `repo:` tag) which writes `BOARD_PR: <url>` into the task result; see Routing above. Note: `agent/**/*.test.mjs` is now in the vitest `include`.
  - **Improvement Scout** (`scout.mjs`, `npm run scout`) is the *idea producer* at the front of the pipeline (the inverse of the dispatcher's *consumer* role): it scans **every repo under `AGENT_REPO_BASE` (~/code)** for high-leverage improvements (infra, devtools, feature, fix, docs — or a brand-new project when the win is large), then **proposes only the single highest-ranked idea** to you on Telegram with inline Yes/No buttons — it queues the task **only if you tap ✅** (tap ❌ or ignore it and nothing runs). It runs a fresh `claude -p ... --output-format json --dangerously-skip-permissions` in the repo base, asking it to brainstorm + score *every* idea on three 1-10 axes (impact, confidence, ease). The scan prompt (`scoutPrompt`) requires each idea's `prompt` to use **repo-relative paths only** (never an absolute `/Users/…/code/<repo>/…`), never `cd`, and **no git steps** — because a task with an embedded absolute path lets the downstream PR-flow agent escape its worktree and edit the real checkout (see the Routing note on `implementPrompt`, which is the backstop for the same failure). **The ranking is computed in code, not trusted from the model** — `iceScore = impact × confidence × ease` (pure, in `lib/scout.mjs`), so it's deterministic and reproducible. `parseScout` (fails closed) pulls the model's `{ ideas: [...] }` JSON block, `rankIdeas`/`selectTop` order them, and `ideaToTask` turns the winner into a task input via three branches: a `repo:` tag (→ default PR route, the common case) when the idea's `repo` resolves to a real dir (separator-insensitively, reusing `matchRepoSlug`); a scaffolding prompt under `repoBase` + `new-project` tag when `category === "new-project"`; else a `workspace` tag with the absolute base prefixed (cross-repo idea, no PR). Every scout task carries a `scout` tag for provenance. **Confirmation flow:** the proposed task is parked as the *single* pending offer in `.data/scout-pending.json` (`lib/pending.mjs`), and the control-plane Telegram bot's `handleCallback` acts on your tap — ✅ POSTs the parked task to the board, ❌ discards it, and the message is stamped with the outcome. **Pause-until-answered:** while a proposal is still within its TTL (`PROPOSAL_TTL_MS`, 24h) the next scheduled run *skips the scan* so unanswered ideas never pile up; once it expires, the next run dismisses the stale offer (stamping the old message) and scans afresh. A tap on a superseded/expired message is rejected (id-matched against the current pending). Pure helpers (`proposalActive`, `proposalKeyboard`, `parseCallback`, `proposalText`) are unit-tested in `lib/scout.test.mjs`; the impure scan + propose live in `scout.mjs`. **Scan memory** (`lib/scout-memory.mjs`, ledger in `.data/scout-memory.json`) does two things so the 2-hourly run is incremental, not a cold re-read of all ~90 repos. **(1) SHA-skip:** each run fingerprints every repo (`git rev-parse HEAD`, marked `+dirty` on uncommitted changes) and **deep-scans only the repos whose fingerprint moved since the last scan, plus any new ones** (`reposToScan`) — unchanged repos are skipped outright, which is the cost saving (on a normal day only a handful of repos changed). **(2) Backlog:** a scan ranks many ideas but only the single best is proposed per run; the rest are persisted as a ranked `ideas` backlog (`mergeBacklog` → `setBacklog`), so a quiet run with **no repo changes skips the model entirely and still proposes from the backlog** — the #2–#5 a prior scan ranked but didn't pitch. Proposing an idea drops it from the backlog (`dropIdea`) so a quiet run advances to the next one, but **only an ACCEPTED idea is permanently suppressed**: ✅ (or the headless direct-queue) records its key in `accepted` (capped) via `recordAccepted` — a ❌/ignore records nothing, so a later scan can **resurface** the idea. The Telegram-propose path stashes the idea's `ideaKey` in the pending record; the bot's `handleCallback` calls `recordAccepted` on ✅ (best-effort — a memory hiccup never fails the queue). Fresh scans dedupe against the backlog and the `accepted` set by a stable `ideaKey` (repo-normalized + title-slug, so near-duplicate phrasings collapse), and fresh ideas for a just-scanned repo supersede that repo's stale backlog entries. `avoidTitles` feeds backlog + accepted titles back into the scan prompt so the model doesn't re-derive them (rejected ideas are deliberately omitted, so it's free to surface them again). A periodic full sweep (`dueForFullScan`, 24h, tunable via `SCOUT_FULL_SCAN_MS`) or `--full` rebuilds the backlog from a fresh all-repo scan, which is also what revisits stagnant unchanged repos. The scoring-aware merge (`ideaKey`/`mergeBacklog`) lives in `lib/scout.mjs` next to `rankIdeas`; the persistence + fingerprinting + `reposToScan`/`recordScan`/`markProposed`/`avoidTitles` in `lib/scout-memory.mjs`. Pure helpers in both are unit-tested (`lib/scout.test.mjs`, `lib/scout-memory.test.mjs`). **It proposes by default** (`--dry-run` previews without proposing, leaving memory untouched; `--print-prompt` dumps the scan prompt; `--full` forces a full sweep). With **Telegram unconfigured it falls back to pushing straight to the board** (old behavior). It needs a board + bot running to accept your ✅. **On-demand trigger:** besides the launchd timer, the control-plane bot fields **`/scout`** (`/scout full` forces a full sweep) — `handleScout` in `telegram-bot.mjs` spawns the same one-shot `scout.mjs` child (via `process.execPath`), acks "scanning…", and on close reports the outcome unless the scout parked a fresh proposal (detected by re-reading pending). Plain `/scout` passes **`--incremental`** so it stays cheap — that flag suppresses the `dueForFullScan` auto-timer, so an on-demand tap never auto-promotes into a full all-repo sweep (the scheduler's 24h timer and `/scout full` own full sweeps; `--incremental` records `full:false`, leaving `lastFullScanAt` untouched, so the due sweep just waits for the next scheduled run). It's guarded: one in-flight `/scout` at a time (module-level `scoutChild`) and it refuses while a proposal is still `proposalActive`. Only works when the control-plane bot owns inbound (`getUpdates`); an external front door on the same token (`--no-telegram`) must add the command itself. Env: `SCOUT_MODEL`, `SCOUT_TIMEOUT` (ms, default 30 min), `SCOUT_FULL_SCAN_MS` (full-sweep interval, default 24h).
  - `launchd/scout-install.mjs` (`npm run scout:install`) writes + loads a macOS LaunchAgent that fires the scout **every 2h during waking hours (08:00–22:00, silent overnight)** via an **array** of `StartCalendarInterval` entries (a one-shot per fire — *not* `KeepAlive`/`RunAtLoad` like the control plane; each run scans/proposes and exits; a calendar array survives reboots and won't drift like a long-sleeping loop). `--start`/`--end`/`--every`/`--minute` retune the window (default `8`/`22`/`2`/`0`), `--dry-run` makes the scheduled runs preview-only, `--print` previews the plist, `--uninstall` removes it; logs to `.data/scout.{out,err}.log`. It runs no board of its own, so the control plane (`npm run agents:install`) must be up to field your tap and accept the queued task. Label `com.davidcjw.agent-task-board.scout`.
- **UI** is in `components/`. Orchestrator `BoardApp.tsx`; drag-and-drop in `Board.tsx`; prompt-first card `TaskCard.tsx` (also renders the agent result block, the dedicated PR-link field, and the compact one-line form for the Done lane). **Send-back**: Review cards get a "Send back for revision" button (`RewindIcon`) → `ReviseDialog` captures a correction note → `BoardApp.handleRevise` applies `revisePatch` (see the Revise flow above). **Archive**: `Task.archivedAt` (set via the `archived` flag on `updateTask` — threaded through both engines + the PATCH route) hides a card from its lane; archived cards render behind a per-lane "show archived" reveal in `Column.tsx`. `lib/board.ts` exposes `isArchived`; `BoardApp` filters archived out of the lanes and lane counts.

## Design system

The visual language is **vendored** from `~/code/design-systems/dragonfly-ds` into `components/ds/` (primitives `Panel`, `Text`, `Button`, `Rule` + `dragonfly.css` tokens). Do not hand-edit `components/ds/dragonfly.css` — re-vendor from source if the DS changes. Fonts (Fraunces / Inter / JetBrains Mono) are self-hosted via `next/font` and mapped onto the `--df-font-*` tokens in `app/globals.css`. Stay faithful to the dark editorial look: black canvas, hairline grid, single orange accent (`#fa4c14`), monospace labels. The four lane status colours live in `app/globals.css` (`@theme`) and `components/status.ts`.

## Conventions

- No emoji as icons — use the inline SVGs in `components/icons.tsx`.
- Tailwind class names must be literal (not interpolated) so JIT picks them up; per-status class maps live in `components/status.ts`.
- Keep it local-first: nothing should make a network request. No analytics, no backend.
